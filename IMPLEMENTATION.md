# Implementation Notes

This document explains the construction of the reflect glass button in the same
pipeline used by the reference implementation: generate shape-specific optical
maps first, then consume them from an inline SVG filter, then layer CSS and DOM
materials on top.

## 1. Shape-Specific Optical Assets

The strongest version of the effect is not a generic procedural filter.

Instead, the repo generates two PNG assets per preset:

1. `displacement-map.png`
   - Encodes the refractive push direction along the rounded shape
   - Built from a rounded-rectangle signed distance field and its gradient
   - Strongest near the rim, softer toward the center

2. `specular-map.png`
   - Encodes rim light, top sheen, and edge glints
   - Also built from the same shape model
   - Used as a highlight carrier inside the SVG filter

These assets are generated by:

- `scripts/generate-reflect-glass-assets.mjs`

Each preset is a contract:

- width
- height
- corner radius
- rim depth

If any of those change materially, generate a new preset instead of only
stretching the old assets with CSS.

## 2. SVG Filter Consumption

Chromium gets the full refraction path through an inline SVG filter:

1. Blur `SourceGraphic` slightly.
2. Read the generated `displacement-map.png` through `feImage`.
3. Refract the sampled backdrop with `feDisplacementMap`.
4. Increase saturation.
5. Read `specular-map.png` through another `feImage`.
6. Mask and blend the specular contribution back over the displaced sample.

This is why the repo keeps preset-specific filters in the demo pages:

- `reflectGlass` for the default pill
- `reflectGlassHero` for the large README close-up pill
- `reflectGlassMotion` for the motion capture pill

Browsers outside the Chromium path still use the same DOM and CSS layers, just
without the asset-backed filter stage.

## 3. Layered DOM And CSS Material

The button itself is built from several stacked layers:

1. Base container
   - Rounded rectangle
   - Transparent shell
   - Main outer shadow
   - Drag transform via CSS variables

2. Surface layer
   - Semi-transparent white gradient
   - `backdrop-filter` blur and saturation
   - In Chromium, also reads `var(--reflect-glass-filter)`

3. Ornament layer
   - Inner translucent shell
   - Helps fake curved thickness and rim depth

4. Top highlight
   - Narrow bright band near the top edge
   - Sized through CSS variables per preset

5. Bottom glow
   - Small blurred accent near the bottom edge
   - Also preset-tuned through CSS variables

The reusable CSS lives in:

- `src/reflect-glass-button.css`

Preset pages only override variables such as:

- `--reflect-glass-min-width`
- `--reflect-glass-min-height`
- `--reflect-glass-radius`
- `--reflect-glass-inner-radius`
- `--reflect-glass-filter`
- highlight/glow tuning values

## 4. Drag Behavior

Dragging is implemented with Pointer Events:

1. On `pointerdown`
   - Store pointer id
   - Store starting pointer coordinates
   - Store current button offset
   - Capture the pointer

2. On `pointermove`
   - Compute delta from the starting pointer position
   - Clamp movement to viewport bounds
   - Update CSS variables `--drag-x` and `--drag-y`

3. On `pointerup` or `pointercancel`
   - Clear drag state
   - Release pointer capture

4. Click suppression
   - If movement exceeds the drag threshold, the next click is prevented
   - This keeps dragging from accidentally triggering navigation

## 5. Initial Placement

The script reads a placeholder anchor element and positions the button relative
to that anchor.

This has two benefits:

- The button starts in a meaningful place without hard-coded viewport numbers
- The same component can be dropped into different layouts

## 6. Integration Contract

To integrate the effect elsewhere while keeping the same construction method,
you need five pieces:

1. Optical assets
   - Generate `displacement-map.png` and `specular-map.png` for the target
     shape

2. Inline SVG filter
   - Use `feImage` to read those generated assets

3. Markup

```html
<div class="reflect-glass-anchor" aria-hidden="true"></div>

<button class="reflect-glass-button" data-reflect-glass type="button">
  <span class="reflect-glass-surface" aria-hidden="true"></span>
  <span class="reflect-glass-ornament" aria-hidden="true"></span>
  <span class="reflect-glass-label">Launch</span>
</button>
```

4. Core stylesheet

- `src/reflect-glass-button.css`

5. Core runtime

- `src/reflect-glass-button.js`

This split matters: the effect is not just one class. It is:

1. generated optical assets
2. a filter wired to those assets
3. layered button markup
4. reusable CSS variables
5. drag/runtime behavior

## 7. Shape Changes Require New Assets

The effect can be resized slightly, but a materially different silhouette should
not reuse the same optical maps.

Examples that should get their own preset:

- larger hero pills
- rounded squares
- circles
- tile-like buttons with different radii

Examples that can usually reuse one preset:

- minor width adjustments
- small label changes
- nearby anchor/position changes

## 8. Why The Demo Is Split From The Core

The repository separates:

- `src/`: reusable implementation
- `demo/`: page layout and presentation only

That separation makes it easier to:

- publish the effect as a snippet or component
- test changes without coupling them to one page design
- embed the button into another system later

## 9. Suggested Open Source Scope

If this example is published independently, the recommended public scope is:

- `assets/`
- `src/`
- `demo/`
- `scripts/generate-reflect-glass-assets.mjs`
- `scripts/serve.mjs`
- `README.md`
- `IMPLEMENTATION.md`
- `LICENSE`

No framework adapters are included by default. Those can be added later as
separate examples if needed.
